## 2. 架构设计

### 2.1 整体架构
采用**三层架构 + 事件驱动**模式：

```
┌─────────────────────────────────────────────┐
│           表现层（Presentation）             │
│  - UI场景（hand_ui, ap_ui, result_panel）   │
│  - 战斗场景（battle_scene, board, unit）     │
└─────────────────┬───────────────────────────┘
                  │ 信号通信
┌─────────────────▼───────────────────────────┐
│            逻辑层（Business Logic）          │
│  - 全局管理器（GameManager, DeckManager）   │
│  - 战斗管理器（CombatResolver, EnemyManager）│
│  - 效果系统（CardEffect 子类）               │
└─────────────────┬───────────────────────────┘
                  │ 数据访问
┌─────────────────▼───────────────────────────┐
│            数据层（Data Layer）              │
│  - 配置资源（CardData, UnitData, Level）     │
│  - 配置加载器（ConfigLoader）                │
│  - 枚举定义（GameEnums）                     │
└─────────────────────────────────────────────┘
```

### 2.2 核心设计原则

#### 2.2.1 事件驱动架构
- **EventBus**：全局事件总线，解耦模块间通信
- **信号机制**：使用Godot的Signal系统进行模块通信
- **观察者模式**：UI监听数据变化自动更新

#### 2.2.2 数据驱动设计
- **Resource配置**：所有游戏数据使用Godot Resource定义
- **热配置**：修改.tres文件即可调整游戏参数
- **配置分离**：逻辑与数据分离，易于扩展和平衡调整

#### 2.2.3 模块化设计
- **职责单一**：每个模块只负责一个功能域
- **低耦合**：模块间通过信号通信，减少直接引用
- **高内聚**：相关功能聚合在同一模块内

## 3. 核心模块设计

### 3.1 全局单例（Autoload）

#### 3.1.1 EventBus（事件总线）
**职责**：全局信号中心，解耦模块间通信

**关键信号**：
```gdscript
# 游戏流程
signal game_started()
signal game_ended(victory: bool)

# 回合流程
signal turn_started(turn_number: int)
signal turn_ended()
signal combat_phase_changed(phase: String)

# 卡牌系统
signal card_drawn(card_data: Resource)
signal card_played(card_data: Resource, target_position: Vector2i)
signal hand_updated(cards: Array)

# 单位系统
signal unit_spawned(unit: Node, position: Vector2i)
signal unit_moved(unit: Node, from_pos: Vector2i, to_pos: Vector2i)
signal unit_died(unit: Node, position: Vector2i)
signal unit_stat_changed(unit: Node, stat_name: String, old: int, new: int)

# 战斗系统
signal combat_started()
signal combat_resolved()
signal enemy_turn_ended()

# 关卡系统
signal level_started(level_config: Resource)
signal victory_achieved()
signal defeat_triggered()

# AP系统
signal ap_changed(current: int, max_value: int)
signal ap_insufficient(required: int, current: int)

# UI系统
signal tile_selected(position: Vector2i)
signal ui_message(message: String, type: String)
```

#### 3.1.2 GameManager（游戏管理器）
**职责**：游戏状态机、回合流转、胜负判定

**核心状态**：
- `GameState`：MENU / BATTLE / SETTLEMENT / GAME_OVER
- `CombatPhase`：PLAYER_TURN / COMBAT / ENEMY_TURN / CHECK_VICTORY

**关键方法**：
- `load_level(level_config)` - 加载关卡
- `start_next_turn()` - 开始下一回合
- `end_player_turn()` - 结束玩家回合
- `trigger_victory() / trigger_defeat()` - 触发胜负
- `is_deployment_turn()` - 判断是否为部署回合

#### 3.1.3 DeckManager（卡组管理器）
**职责**：牌库管理、抽卡、打出卡牌

**核心数据**：
```gdscript
var library: Array[Resource]           # 永久牌库（跨关卡）
var is_library_initialized: bool = false  # 牌库是否已初始化
var draw_pile: Array[Resource]         # 抽卡堆
var hand: Array[Resource]               # 手牌
var discard_pile: Array[Resource]      # 弃牌堆
```

**关键方法**：
- `initialize_deck(starter_cards)` - 初始化牌库（仅首次）
- `prepare_battle_deck()` - 准备战斗牌组
- `draw_cards(count)` - 抽牌
- `draw_summon_cards_only(count)` - 只抽召唤卡（第0回合）
- `play_card(card, target)` - 打出卡牌
- `add_card_to_library(card)` - 添加卡牌到永久牌库

#### 3.1.4 ConfigLoader（配置加载器）
**职责**：加载和缓存所有配置资源

**加载路径**：
```gdscript
const BATTLE_RULES_PATH = "res://resources/configs/default_rules.tres"
const CARDS_DIR = "res://resources/cards/"
const UNITS_DIR = "res://resources/units/"
const LEVELS_DIR = "res://resources/configs/levels/"
```

**缓存机制**：
- 启动时加载所有配置到内存
- 通过ID快速查询
- 支持热重载（修改.tres文件后重新加载）

### 3.2 战斗系统模块

#### 3.2.1 BattleController（战斗控制器）
**职责**：战斗场景总控制，协调各子系统

**子系统引用**：
- Board（棋盘）
- APManager（AP管理）
- CombatResolver（战斗结算）
- EnemyManager（敌人管理）
- VictoryChecker（胜负判定）

**核心流程**：
```gdscript
func _ready():
    # 1. 加载关卡
    load_test_level()
    # 2. 初始化牌库（仅首次）
    DeckManager.initialize_deck(starter_cards)
    # 3. 开始第0回合
    EventBus.turn_started.emit(0)

func _on_turn_started(turn_number):
    # 更新UI
    # 生成敌人波次
    spawn_enemy_waves(turn_number)

func _on_end_turn_pressed():
    # 触发回合结束
    EventBus.turn_ended.emit()
```

#### 3.2.2 Board（棋盘）
**职责**：棋盘网格管理、单位部署和移动

**数据结构**：
```gdscript
var tiles: Array[Array]                # 二维数组存储Tile
var units: Dictionary                  # {Vector2i: Unit} 单位位置映射
var rows: int = 3
var cols: int = 10
```

**关键方法**：
- `initialize_board(battle_rules)` - 初始化棋盘
- `spawn_unit(unit_data, position)` - 生成单位
- `move_unit(from, to)` - 移动单位
- `get_unit_at(position)` - 查询单位
- `get_units_by_faction(faction)` - 按阵营获取单位
- `set_tiles_selectable(positions)` - 设置格子可选状态

#### 3.2.3 Unit（单位）
**职责**：单位状态管理、视觉更新

**核心属性**：
```gdscript
var unit_data: UnitData                # 单位模板数据
var faction: GameEnums.Faction         # 阵营
var max_hp: int                        # 最大生命值
var current_hp: int                    # 当前生命值
var attack: int                        # 攻击力
var armor: int                         # 护甲
var corruption_count: int = 0          # 腐化计数
```

**关键方法**：
- `take_damage(damage, source)` - 受到伤害
- `apply_corruption()` - 应用腐化
- `corrupt_to_enemy()` - 转化为敌人
- `update_visual()` - 更新显示

#### 3.2.4 CombatResolver（战斗结算）
**职责**：执行战斗阶段的攻击判定

**攻击流程**：
```gdscript
func execute_combat():
    # 1. 玩家单位攻击
    execute_faction_attacks(GameEnums.Faction.PLAYER)
    # 2. 敌人单位攻击
    execute_faction_attacks(GameEnums.Faction.ENEMY)
    # 3. 发出战斗结束信号
    EventBus.combat_resolved.emit()
```

**攻击逻辑**：
- 近战：攻击同行首个敌人
- 远程：待扩展
- 伤害计算：优先扣除护甲，剩余扣生命

#### 3.2.5 EnemyManager（敌人管理器）
**职责**：敌人推进、腐化判定

**推进逻辑**：
```gdscript
func execute_enemy_push():
    var enemies = board.get_units_by_faction(ENEMY)
    # 按X坐标排序（左到右）
    enemies.sort_custom(...)
    
    for enemy in enemies:
        var target_pos = current_pos - Vector2i(1, 0)
        if target_pos.x < 0:
            # 边界警告
            continue
        
        var target_unit = board.get_unit_at(target_pos)
        if target_unit:
            if target_unit.faction == PLAYER:
                # 腐化判定
                target_unit.apply_corruption()
            # 受阻，停留原位
        else:
            # 正常推进
            board.move_unit(current_pos, target_pos)
```

#### 3.2.6 VictoryChecker（胜负判定器）
**职责**：检查胜利和失败条件

**判定时机**：监听 `CHECK_VICTORY` 阶段

**失败条件**：
```gdscript
func check_defeat() -> bool:
    var enemies = board.get_units_by_faction(ENEMY)
    for enemy in enemies:
        if enemy.grid_position.x <= 0:
            return true  # 敌人到达第0列
    return false
```

**胜利条件**：
```gdscript
func check_victory() -> bool:
    # 1. 所有波次回合已过
    var all_waves_passed = current_turn > last_wave_turn
    # 2. 场上无敌人
    var no_enemies = enemies.is_empty()
    return all_waves_passed and no_enemies
```

#### 3.2.7 APManager（行动点管理器）
**职责**：AP管理、消耗和恢复

**核心逻辑**：
```gdscript
var current_ap: int = 0
var ap_per_turn: int = 3
var max_ap: int = 999  # 无上限

func _on_turn_started(turn_number):
    # 每回合重置为固定AP
    current_ap = ap_per_turn
    EventBus.ap_changed.emit(current_ap, max_ap)

func spend_ap(amount) -> bool:
    if current_ap < amount:
        EventBus.ap_insufficient.emit(amount, current_ap)
        return false
    current_ap -= amount
    EventBus.ap_changed.emit(current_ap, max_ap)
    return true
```

#### 4.2.2 CardEffect（卡牌效果系统）
**基类设计**：`scripts/data/card_effect.gd`

```gdscript
class_name CardEffect
extends Resource

# 效果上下文
class EffectContext:
    var caster              # 施法者
    var target_position     # 目标位置
    var target_unit         # 目标单位
    var board               # 棋盘引用

# 执行效果（子类重写）
func execute(context: EffectContext) -> void:
    pass
```

**效果实现类**：

1. **SpawnUnitEffect（召唤单位效果）**

     - `value: int`（增益数值）
     - `duration: int`（持续时间，-1为永久）
   - 执行：修改目标单位属性

```ini
    "undead_armor:5"
```

#### 4.5.2 奖励生成逻辑

```gdscript
func generate_rewards() -> Array:
    var all_entries = []
    
    # 解析卡牌池
    for entry in card_pool:
        var parts = entry.split(":")
        all_entries.append({
            "type": "card",
            "id": parts[0],
            "weight": int(parts[1])
        })
    
    # 加权随机抽取
    var rewards = []
    for i in reward_count:
        var reward = weighted_random(all_entries)
        rewards.append(reward)
    
    return rewards
```

#### 4.5.3 配置奖励池流程

1. **在关卡配置中定义SubResource**：
   - 设置 `reward_count`（推荐3个）
   - 添加卡牌到 `card_pool`，格式：`"card_id:权重"`

2. **权重设计原则**：
   - 普通卡牌：权重10
   - 稀有卡牌：权重5
   - 权重越高，出现概率越大

```
    ↓

    ↓

    ↓

    ↓
```

## 7. 性能优化与最佳实践

### 7.1 性能考虑

1. **对象池模式**：
   - 单位、卡牌按钮可考虑使用对象池
   - 减少频繁的实例化和销毁

2. **信号解绑**：
   - 节点销毁时确保信号断开
   - 使用 `queue_free()` 而不是 `free()`

3. **配置缓存**：
   - ConfigLoader启动时加载所有配置
   - 避免运行时频繁加载Resource

### 7.2 代码规范

1. **命名规范**：
   - 类名：PascalCase（CardData）
   - 变量/函数：snake_case（current_hp）
   - 常量：UPPER_SNAKE_CASE（COLOR_NORMAL）
   - 信号：snake_case（turn_started）

2. **注释规范**：
   - 类级注释：说明职责
   - 函数注释：说明参数和返回值
   - 避免无意义注释

3. **文件组织**：
   - 一个文件一个类
   - 相关文件放在同一目录
   - Resource配置按类型分目录

### 7.3 调试技巧

1. **日志系统**：
   - 使用统一的日志前缀：`[ModuleName]`
   - 重要状态变化打印日志
   - 生产环境可关闭调试日志

2. **配置验证**：
   - 启动时检查必需配置是否存在
   - 使用 `push_error` 和 `push_warning`

3. **可视化调试**：
   - 使用 Godot Remote调试器
   - 监控信号发射
   - 查看场景树和节点属性

    ```
    effect_001 | buff_stat | armor | 5 | -1  # 同一EffectID，多个效果
    ```
